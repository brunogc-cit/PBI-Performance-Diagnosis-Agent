The ADE - Trade semantic model (45 tables, 801 measures, 31 relationships) drives 1.24 million daily queries against Databricks, with 5.1% exceeding 10 seconds and 0.8% exceeding 30 seconds. The main fact table (fact_product_option_trade_daily_snapshot_v1, ~116.5 GB, DirectQuery) is a hub with degree 21 and generates the bulk of query load at 42,764 queries/day. The slowest queries read up to 6 billion rows (19.6 GB) to return fewer than 5,000 rows — a scan amplification ratio exceeding 1,000,000:1. Real PBI-generated SQL confirms all 91 columns are selected from the fact table even when visuals need only 1-2 aggregated values. 49 critical-complexity DAX measures generate 14+ SQL subqueries each via nested CALCULATE and FILTER(ALL) patterns. 77 measures use FILTER(ALL) which forces full table scans. The #1 recommendation is to create a narrow serve view with only the ~23 columns referenced by PBI, combined with refactoring the top FILTER(ALL) measures to use KEEPFILTERS or explicit column filters.

F1: Column Over-Selection on Fact Table (91 cols vs ~23 needed)

critical Model-wide dbt Models Semantic Model

Problem: PBI DirectQuery selects all 91 columns from sales.serve.fact_product_option_trade_daily_snapshot_v1 even when a visual only needs 1-2 aggregated values (e.g. stock_quantity). The serve view is a direct SELECT * from the curated table with no column reduction. This generates 42,764 queries/day reading an average of 2.5s each, with P95 at 10s. The slowest queries read 19.6 GB per execution.

Affected Objects:
Tables: fact_product_option_trade_daily_snapshot_v1
dbt Models: serve_fact_product_option_trade_daily_snapshot_v1

How to Fix

Create a narrow serve view serve_fact_trade_daily_narrow_v1 in dbt containing only the ~23 columns actually referenced by PBI measures: dim_date_sk, dim_product_option_sk, dim_pricing_store_sk, dim_web_store_sk, dim_shop_sk, dim_warehouse_sk, dim_country_sk, dim_territory_sk, dim_selling_channel_sk, dim_seller_sk, dim_price_status_sk, dim_price_status_group_sk, dim_inventory_status_sk, dim_product_lifecycle_status_sk, dim_lifecycle_price_status_sk, dim_stock_age_band_sk, financial_year_week_id, plus the measure columns (retail_sale_quantity, retail_sale_value, stock_quantity, etc.). Then update the PBI partition expression to point to the new view.

Estimated improvement: 60-75% reduction in read_bytes per query. Query time reduction from avg 2.5s to ~0.8-1.2s.

Impact: high Effort: low
Trade-offs:
  • Requires dbt deploy cycle to create new view
  • PBI partition expression update needed
  • Must ensure all DAX measure references are included in narrow view

F2: FILTER(ALL) Anti-Pattern in 77 Measures Forcing Full Table Scans

critical Model-wide Semantic Model

Problem: 77 DAX measures use FILTER(ALL(...)) which overrides all filters and forces DirectQuery to scan the entire fact table on every evaluation. Combined with context transitions (avg 7-8 per critical measure), the top 49 critical-complexity measures generate an estimated 14 SQL subqueries each. This is the primary driver of the 5-6 billion rows read in the slowest queries.

Affected Objects:
Measures: Retail Sales Units Coversion % LY, Retail Sales Value OB LM, Retail Sales Value T&R LM, Intake Units WTD, Retail Physical Return Quantity WTD, Retail Physical Return Value WTD, Retail Return Value WTD, Retail Sales Quantity WTD, Retail Sales Value LY WTD, Retail Sales Value WTD (+2 more)

How to Fix

Refactor the top 20 FILTER(ALL) measures to use KEEPFILTERS or explicit column-level filters. For WTD/MTD/HTD/YTD measures, replace FILTER(ALL(Date), Date[financial_year_week_id] = ...) with CALCULATE([Base Measure], KEEPFILTERS(Date[financial_year_week_id] = ...)). For ratio measures like 'Retail Sales Units Conversion % LY', use DIVIDE with a pre-computed denominator variable. Prioritise the 49 critical-complexity measures first.

Estimated improvement: 50-80% reduction in rows scanned for affected measures. P95 query time reduction from 10.2s to ~3-5s.

Impact: high Effort: high
Requires Deep Dive
  1. Export top 20 FILTER(ALL) measures and run in DAX Studio with Server Timings
  2. Identify which FILTER(ALL) patterns can be replaced with KEEPFILTERS
  3. Test WTD measures with explicit time intelligence filters
Trade-offs:
  • DAX refactoring requires careful testing of each measure to ensure functional equivalence
  • Some FILTER(ALL) patterns may be intentional for ratio calculations

F3: Date Filter via JOIN Instead of Direct WHERE on Fact Table

high Model-wide Semantic Model dbt Models

Problem: PBI applies date predicates by JOINing dim_date_v2 with WHERE calendar_date = '2026-04-12', then joining to the fact table via dim_date_sk surrogate key. Databricks cannot push this filter down to the fact table's liquid clustering on dim_date_sk because the predicate is on a column in a different table. The fact table (116.5 GB) is scanned without cluster pruning, reading billions of unnecessary rows.

Affected Objects:
Tables: Daily Trade, Date
dbt Models: serve_fact_product_option_trade_daily_snapshot_v1, serve_dim_date_v2
Relationships: Daily Trade → Date (dim_date_sk)

How to Fix

Option A: Add a calculated column to the fact table with the actual calendar_date (or a calendar_date filter column) and create a direct relationship, enabling PBI to filter directly on the fact. Option B: Create a parameterised M query that applies a date WHERE clause before the fact table is loaded. Option C: Add a WHERE filter on the serve view using a date range parameter. The clustering on dim_date_sk is present but ineffective because PBI never sends a direct predicate on this column.

Estimated improvement: 90%+ row reduction on date-filtered queries. Queries that read 4.4B rows should drop to ~10-50M rows.

Impact: high Effort: medium
Connection Mode Comparison:
ModeImplications
DirectQueryCurrent — date filter via JOIN, no cluster pruning, 4.4B rows scanned
HybridAggregation table for common date ranges in Import mode, DQ fallback for ad-hoc dates
Parameter FilterM-level date parameter reduces fact scan to current period only
Trade-offs:
  • Calculated column approach increases model complexity
  • M-level filtering requires report authors to use the date parameter

F4: 3 Bidirectional Cross-Filter Relationships on DirectQuery Tables

high Model-wide Semantic Model

Problem: Three bidirectional cross-filter relationships exist in the model. In DirectQuery mode, bidirectional filtering causes additional SQL subqueries because Databricks must evaluate the filter in both directions. The Option table (dual, degree 9) is involved in all three, amplifying the impact.

Affected Objects:
Relationships: Option ↔ Option Ranking, Option ↔ Product Option In Pricing Store, Option ↔ Product Option In Store

How to Fix

Use CROSSFILTER(Option[dim_product_option_sk], 'Product Option In Store'[dim_product_option_sk], BOTH) in specific measures instead of model-level bidirectional.

Estimated improvement: 10-20% reduction in SQL subquery count for queries touching these relationships.

F5: 49 Critical-Complexity DAX Measures Generating 14+ SQL Subqueries

high Model-wide Semantic Model

Problem: 49 measures have critical complexity (score 44-45) driven by 7-8 context transitions and 5-6 FILTER(ALL) calls each. Each generates an estimated 14 SQL subqueries in DirectQuery mode. These are predominantly WTD/MTD/HTD/YTD time intelligence measures that nest multiple CALCULATE layers with FILTER(ALL(Date)).

Affected Objects:
Measures: Retail Sales Units Coversion % LY, Retail Sales Value OB LM, Retail Sales Value T&R LM, Intake Units WTD, Retail Physical Return Quantity WTD, Retail Physical Return Value WTD, Retail Return Value WTD, Retail Sales Quantity WTD, Retail Sales Value LY WTD, Retail Sales Value WTD

How to Fix

Consolidate time intelligence into shared base measures. Create a single WTD base measure using CALCULATE with KEEPFILTERS, then reference it from all WTD variants (Retail Sales Value WTD, Retail Return Value WTD, etc.). This reduces context transitions from 7-8 to 2-3 per measure. Use VAR/RETURN pattern to avoid repeated subexpression evaluation.

Estimated improvement: 60-70% reduction in SQL subquery count for time intelligence measures.

Impact: high Effort: high
Requires Deep Dive
  1. Group the 49 critical measures by pattern (WTD, MTD, HTD, YTD, LY)
  2. Create a template refactoring for each pattern
  3. Test in DAX Studio with Server Timings before and after
Trade-offs:
  • Large refactoring effort — 49 measures to update
  • Functional equivalence must be verified for each

F8: Query Reduction Settings Not Optimised on Trade Report

medium Report-specific PBI Report

Problem: The Trade report has cross-highlighting enabled and slicer/filter 'Apply' buttons disabled. In DirectQuery mode, cross-highlighting generates additional queries each time a user hovers over a visual (the engine must re-query related visuals). Without 'Apply' buttons, every slicer or filter selection immediately fires queries instead of batching them. Auto-refresh is NOT enabled — this is a query reduction settings issue, not an auto-refresh issue.

Affected Objects:
Pages: Trade report (report-level query reduction settings)

How to Fix

In PBI Desktop → File → Options → Report settings → Query reduction: (1) Disable cross-highlighting to prevent hover-triggered queries. (2) Enable 'Add an Apply button to each slicer' so users can select multiple slicer values before queries fire. (3) Enable 'Add an Apply button to each filter' for the same reason. The 'Apply all slicers at once' option is already enabled, which is good.

Estimated improvement: 5-15% reduction in interactive query volume during user sessions.

Impact: medium Effort: low

F10: 8 Dual Mode Tables That Could Be Import

medium Model-wide Semantic Model

Problem: 8 financial transform/time intelligence tables are in Dual mode but contain relatively static reference data (financial calendar mappings). In Dual mode, PBI uses DirectQuery when the table is joined with a DQ table, adding unnecessary JOINs to Databricks queries. Since these are small lookup tables, switching to Import would cache them locally and eliminate Databricks roundtrips.

Affected Objects:
Tables: Daily Financial Transform, Daily Financial Half to Date, Daily Financial Month to Date, Daily Financial Year to Date, Weekly Financial Transform, Weekly Financial Half to Date, Weekly Financial Month to Date, Weekly Financial Year to Date

How to Fix

Switch the 8 financial transform tables to Import mode. They contain static calendar/financial period mappings that change infrequently. Set a scheduled refresh (e.g., daily at 06:00). This removes 8 tables from DirectQuery SQL generation.

Estimated improvement: 5-10% reduction in query complexity by removing financial lookup table JOINs from Databricks SQL.

Impact: medium Effort: low

Impact: medium Effort: low

F12: 6 Engineering BPA Findings: Functions on Filter Columns (E05)

medium Model-wide dbt Models Semantic Model

Problem: 6 dbt models apply functions on columns that are used in WHERE or JOIN predicates (e.g., CONCAT_WS on surrogate keys, CAST on date columns). Functions on filter columns prevent predicate pushdown and disable clustering benefits.

Affected Objects:
dbt Models: 6 dbt models with functions applied on filter/join columns

How to Fix

Review the 6 flagged models and remove function wrapping on filter/join columns. For computed composite keys (like dim_product_option_in_pricing_store_sk = concat_ws('|', ...)), consider pre-computing in the curated layer instead.

Estimated improvement: 5-15% improvement in cluster pruning efficiency for affected tables.

Impact: medium Effort: medium

F13: 54 Nested CALCULATE Patterns in DAX Measures

medium Model-wide Semantic Model

Problem: 54 measures contain nested CALCULATE expressions. In DirectQuery, each CALCULATE nesting generates an additional level of SQL subquery. Combined with FILTER(ALL), these create the 4-5 level deep subquery patterns observed in real Databricks query history.

Affected Objects:
Measures: 54 measures with nested CALCULATE (each nesting adds a SQL subquery level)

How to Fix

Flatten nested CALCULATE by using VAR/RETURN pattern. Instead of CALCULATE(CALCULATE([Measure], filter1), filter2), use VAR filtered = CALCULATE([Measure], filter1, filter2) RETURN filtered. This reduces subquery depth.
